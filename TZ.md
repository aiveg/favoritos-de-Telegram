
## Цель проекта

Сервис на Python, который раз в час забирает новые сообщения из Saved Messages в Telegram через Telethon, сохраняет медиафайлы на диск и метаданные в SQLite, а затем отдаёт их через локальный веб-сервер с динамической пагинацией, сортировкой, фильтрацией и поиском — без хранения полного HTML со всеми сообщениями. [stackoverflow](https://stackoverflow.com/questions/78161842/using-telethon-to-read-messages-from-a-particular-group)

## Общая архитектура

Два независимых процесса: воркер-сборщик (работает по cron/systemd timer раз в час) и веб-сервер (работает постоянно, слушает localhost, рендерит страницы по запросу из базы данных). [geeksforgeeks](https://www.geeksforgeeks.org/python/how-to-implement-filtering-sorting-and-pagination-in-flask/)

- Сборщик: Python + Telethon, инкрементальная выгрузка новых сообщений из `client.iter_messages('me')`.
- Хранилище файлов: файловая система, папка `media/`, подпапки по типам контента.
- Хранилище метаданных: SQLite (не JSON) — обязательна для быстрых запросов, фильтрации, сортировки и полнотекстового поиска.
- Веб-сервер: Flask или FastAPI, отдаёт HTML через Jinja2-шаблоны с server-side pagination. [youtube](https://www.youtube.com/watch?v=kYRvhQj-JQE)
- Конфигурация: файл `.env` (api_id, api_hash, session_name, путь к папке медиа, путь к БД, порт веб-сервера, часовой пояс).

## Модуль 1: сборщик (collector.py)

Задача — раз в час подключаться к Telegram под личным аккаунтом и докачивать всё новое из избранного.

- Авторизация по api_id/api_hash из .env, сессия сохраняется в файл (`.session`), чтобы не логиниться повторно.
- При каждом запуске брать `MAX(message_id)` из БД и запрашивать `iter_messages('me', min_id=last_id, reverse=True)` — тянуть только новые сообщения.
- Определять тип контента по атрибутам: фото, видео, круглое видео (`DocumentAttributeVideo.round_message=True`), голосовые (`DocumentAttributeAudio.voice=True`), обычное аудио, документы, стикеры, кастомные эмодзи (`DocumentAttributeCustomEmoji`), GIF, текст, альбомы (grouped_id).
- Скачивать файл в `media/<тип>/<message_id>_<original_filename или hash>`, избегать повторного скачивания уже существующих файлов.
- Записывать в таблицу SQLite: message_id, дата, тип контента, текст/caption, путь к файлу, размер файла, длительность (для видео/аудио), grouped_id (для альбомов), список хештегов/сущностей из текста.
- Логировать ошибки скачивания в отдельный лог-файл, не прерывать весь батч из-за одного сбойного файла.
- Обрабатывать rate-limit (FloodWaitError) с ожиданием и повтором.

## Модуль 2: веб-сервер (server.py)

Задача — рендерить страницы по запросу, а не генерировать статический HTML со всем содержимым.

- Роут `/` — главная страница, читает параметры из query string (page, per_page, sort, order, filter_type, date_from, date_to, search) и отдаёт только нужную порцию записей из БД через `SELECT ... LIMIT/OFFSET` или `keyset pagination`. [stackoverflow](https://stackoverflow.com/questions/58771236/pagination-for-sqlite)
- Сортировка: по дате (новые/старые), по типу контента, по размеру файла.
- Фильтрация: по типу контента (фото/видео/аудио/документ/кружочек/стикер/кастом-эмодзи/текст), по диапазону дат, по наличию текста/caption.
- Поиск: полнотекстовый по тексту сообщения и caption, использовать SQLite FTS5 для скорости на больших объёмах.
- Каждая карточка — миниатюра/превью (для фото/видео генерировать thumbnail при скачивании, чтобы не грузить оригинал), клик открывает полный файл.
- Пагинация — классические номера страниц + "следующая/предыдущая", количество на странице настраивается через .env или UI.
- Все ответы — обычный HTML (SSR через Jinja2), без необходимости JS-фреймворка; допустимо немного JS для AJAX-подгрузки без перезагрузки страницы (опционально).

## Схема базы данных (минимум)

| Таблица | Поля |
|---|---|
| messages | id, message_id, date, content_type, text, file_path, thumbnail_path, file_size, duration, grouped_id, width, height |
| tags (опционально) | message_id, tag |

## Требования к конфигурации (.env)

- TG_API_ID, TG_API_HASH, TG_SESSION_NAME
- MEDIA_DIR, DB_PATH
- SERVER_HOST, SERVER_PORT, ITEMS_PER_PAGE
- TIMEZONE, LOG_LEVEL

## Дополнительные функции (что ещё добавить)

- Экспорт результатов фильтра в zip-архив одним кликом.
- Просмотр альбомов (grouped_id) как единой карусели, а не отдельными карточками.
- Тег-облако/автотеги по хэштегам из текста сообщений, отдельный фильтр по тегам.
- Дедупликация файлов по хэшу (если одно и то же изображение сохранено дважды).
- Простая аутентификация (basic auth) на веб-сервере, если он будет доступен не только с localhost.
- Отдельный dashboard-виджет со статистикой: сколько файлов какого типа, общий объём, график сохранений по дням.
- Watch-режим: вместо cron — постоянно работающий воркер с `client.run_until_disconnected()` и обработчиком новых сообщений в реальном времени, без ожидания часа.
- Резервное копирование БД и медиа на удалённый storage (S3-совместимый) по расписанию.

## Нефункциональные требования

- Идемпотентность: повторный запуск сборщика не должен дублировать записи или файлы.
- Устойчивость к разрыву соединения и большим объёмам данных (тысячи сообщений).
- Совместимость с запуском на macOS (launchd/cron) и Ubuntu (systemd timer/cron) без изменения кода — весь путь и специфика окружения только через .env.
- Логирование всех операций (скачивание, ошибки, запуск сервера) в файл с ротацией.
